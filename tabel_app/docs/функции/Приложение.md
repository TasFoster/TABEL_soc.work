# Функция «Приложение к табелю»

Формирует помесячное **приложение к табелю** в формате `.xls`: матрица нагрузки
**чел/день** на каждого соцработника по двум секторам с расчётом начисления и
**перераспределением нагрузки при отсутствии**. Запись `.xls` — с нуля через `xlwt`,
Microsoft Excel не требуется.

- **Пакет:** `app/features/prilozhenie/`
- **Ключ функции:** `prilozhenie`
- **Секторы:** `gor` (город, «гор.») и `chast` (частный, «част»).
- **Выход:** `.xls` в альбомной ориентации, вписанный в 1 страницу; два связанных блока —
  слева подневная сетка, справа сводка с нормами и начислением.

---

## Поток данных

```
GUI (отделение, месяц, год; правка нагрузки/периодов/отсутствий/перераспределения)
  └► service.compute_month ──► calc.compute ──► service.build_context ──► excel_writer.generate ──► .xls
        (хранилище pril_*)        (бизнес-логика)
```

## Слои

### GUI (`gui.py`) — `PrilozhenieWindow`

- Выбор **отделения / месяца / года**; при изменении — пересчёт (`_recompute` →
  `service.compute_month`).
- Treeview соцработников: №, ФИО, город/день, частный/день, отсутствия, всего город,
  всего частн.; подсветка превышения порога (тег `over`) и неактивных (`inactive`).
- Диалоги:
  - **`LoadEditor`** — постоянная нагрузка и нормы соцработника + флаг активности →
    `pril_load_default`.
  - **`PeriodsEditor`** — помесячные переопределения нагрузки блоками дат по сектору →
    `pril_period`.
  - **`AbsenceDialog`** — периоды отсутствий с кодом → `pril_absence`.
  - **`RedistributionDialog`** — ручное перераспределение чел/дней (есть авто-черновик для
    нагрузки отсутствующих) → `pril_redistribution`.
- Кнопка **«Сформировать приложение»** → `generate_prilozhenie`.

### Сервис (`service.py`)

- `compute_month(dept_id, year, month)` → собирает отделение, активных соцработников,
  рабочие дни (из `ProductionCalendar`), периоды/отсутствия/перераспределения и зовёт
  `calc.compute`; возвращает `(dept, workers, result)`.
- `build_context(...)` — формирует словарь для писателя (заголовки, ndays, рабочие дни,
  подневные итоги, общие итоги, предупреждения).
- `generate_prilozhenie(...)` — высокоуровневый экспорт в `.xls`.

### Хранилище (`storage.py`)

Обёртка над `core.db`; таблицы функции (все с `employee_id … ON DELETE CASCADE`):

| Таблица | Что хранит | Методы |
|---|---|---|
| `pril_load_default` | постоянная нагрузка/нормы/активность (1 строка на сотрудника) | `get_workers`, `save_worker_load` |
| `pril_period` | помесячные переопределения нагрузки блоками дат | `get_periods`, `save_periods` |
| `pril_absence` | отсутствия (диапазон дней + код) | `get_absences`, `save_absences` |
| `pril_redistribution` | передача чел/дней между сотрудниками по сектору | `get_redistributions`, `save_redistributions` |

`get_workers` присоединяет нагрузку из `pril_load_default` и перенумеровывает активных.

### Модель (`model.py`)

- Константы: `SECTORS = ("gor", "chast")`, `SECTOR_LABEL`, `THRESHOLD = {"gor": 20, "chast": 16}`.
- Датаклассы: `Worker` (нормы по умолчанию: gor 10, chast 8), `Period`, `Absence`,
  `Redistribution`.

### Расчёт (`calc.py`) — ядро бизнес-логики

`compute(workers, working_days, ndays, periods, absences, redistributions)` строит
`day_value[(emp_id, sector)][day]` в 4 шага:

1. **База:** на каждый рабочий день — постоянная нагрузка сектора.
2. **Периоды:** строки `Period` замещают значение на своём диапазоне дат (для своего сектора).
3. **Отсутствия:** на дни отсутствия обнуляются **оба сектора** (сотрудник выбывает целиком).
4. **Перераспределение:** на диапазоне дней `value` вычитается у `from` (не ниже 0) и
   добавляется `to` — **в пределах одного сектора**.

Затем считаются: итоги по сектору, подневные итоги по всем, общий итог, **начисление**
`nachisleno[sector] = (oklad / (norma[sector] × число_рабочих_дней)) × итог[sector]`,
и предупреждения о превышении порога (`THRESHOLD`).

### Писатель `.xls` (`excel_writer.py`)

Строится **с нуля** (без шаблона), Calibri 11. Стили кэшируются по сигнатуре
(`_Styles.get`), объединения через `write_merge`, выходные заливаются серым
`SHADE_WEEKEND = 0xE6E6E6`. Печать — альбомная, `fit_width_to_pages = 1`.

**Раскладка (1-based колонки, две строки на соцработника — gor/chast):**

| Блок | Колонки |
|---|---|
| Слева | `L_NUM`=№, `L_FIO`=ФИО, `L_SEC`=сектор, `L_DAY1…`=дни (4…3+ndays), `L_TOTAL`=всего |
| Справа (через пустую колонку) | №, ФИО, сектор, `R_NORMA`, `R_TOTAL`, `R_OKLAD`, `R_NORMACD` (норма×рабочие дни), `R_NACH` (начислено) |
| Строки | `ROW_TITLE`/`ROW_BRANCH`/`ROW_OSOD`/`ROW_HDR`/`ROW_DAYNUM`(числа дней), `FIRST_WORKER_ROW`=6 |

В конце — две строки «Итого город» / «Итого частный».

## Бизнес-правила и краевые случаи

- **Отсутствие обнуляет оба сектора** одновременно (см. шаг 3).
- **Перераспределение посекторно** (город→город, частный→частный); донор не уходит ниже 0.
- **Периоды переопределяют** постоянную нагрузку только на своём диапазоне; вне — берётся
  постоянная.
- Нормы по умолчанию: город 10, частный 8 чел/день. Пороги предупреждения: город >20,
  частный >16.
- Заведующие («зав») автоматически помечаются неактивными при импорте состава (`db.py`),
  в приложение не попадают.
- Удаление сотрудника каскадно уносит все его `pril_*` строки.

## Заметки разработчику

- **Ловушка Tkinter в диалогах (`gui.py`).** Классы-диалоги наследуют `tk.Toplevel`, у
  которого в `__init__` создаётся **строковый атрибут экземпляра `self._name`** (имя
  виджета). Поэтому **нельзя** объявлять в таких классах метод с именем `_name` — атрибут
  перекроет метод, и вызов упадёт с `TypeError: 'str' object is not callable`. Из-за этого
  ранее ломалось перераспределение: `RedistributionDialog._refresh()` вызывал `self._name(...)`
  при непустом списке переносов (открытие диалога с сохранёнными строками, а также сразу
  после «Добавить вручную»/«Авто-черновик»), диалог падал и ничего не пересчитывалось.
  Метод переименован в `_emp_name`. При добавлении методов в диалоги избегать имён,
  совпадающих с атрибутами Tkinter (`_name`, `_w`, `master`, `children`, `tk`).

# Функция «Реестр»

Формирует **реестр оплаты** (РЕЕСТР) в формате `.ods` из входных Excel-реестров,
сгруппированный по соцработникам. Запись `.ods` — на чистом Python через `odfpy`,
Microsoft Excel не требуется.

- **Пакет:** `app/features/reestr/`
- **Ключ функции:** `reestr`
- **Вход:** до трёх `.xls/.xlsx` + **необязательный** журнал:
  - реестр **государственных** услуг (формат `_054`);
  - реестр **дополнительных** услуг (формат `_055`);
  - **отчёт ИПСУ** (формат `402`);
  - **журнал договоров** — «Отчёт по количеству заключённых договоров» (необязательно;
    для отметок новый/пересмотр/снят — см. ниже).
- **Выход:** один файл `РЕЕСТР…ods` с **4 листами**: `Гос_`, `Доп`, `деньги`, `пересмотр`.

---

## Поток данных

```
3 входных .xls ──► parser.py ──► service.prepare() ──► (нераспределённые клиенты?)
                                       │                        │
                                       │                        ▼
                                       │                AssignDialog (назначить соцработника)
                                       ▼
                       registry_builder.build() ──► ods_writer.generate() ──► РЕЕСТР.ods
```

1. Пользователь выбирает папку с входными файлами (или указывает каждый вручную).
2. GUI **автоопределяет** тип каждого файла по ключевым словам в имени, а при
   неоднозначности — по «десятичной доле» сумм (`gui._detect`, `_decimal_fraction`).
3. `service.prepare()` парсит файлы и сверяет клиентов с сохранённой группировкой.
4. Новые (нераспределённые) клиенты → диалог назначения соцработника.
5. `registry_builder.build()` строит иерархию по соцработникам с отметками
   «новый»/«пересмотр».
6. `ods_writer.generate()` клонирует строки шаблона и заполняет 4 листа.

## Слои

### GUI (`gui.py`) — `ReestrWindow`

- Выбор папки/файлов; автоопределение типов (`_detect`, `_pick_folder`).
- Combobox отделения (из `settings.json`: `{number, zav_fio}`).
- Кнопка **«Подготовить»** → `_prepare()`; **«Сформировать РЕЕСТР (.ods)»** → `_generate()`.
- **`AssignDialog`** — модальное назначение соцработника каждому новому клиенту
  (с подсказками из колонки «соцработник» исходного реестра, «Принять все подсказки»).
- **`ClientsManager`** — постоянный редактор групп: список соцработников (порядок ↑/↓,
  добавить/удалить) и их клиентов (добавить/удалить/переименовать/переназначить);
  сохраняется в `app.db` через `storage`.

### Сервис (`service.py`)

- `prepare(gos, dop, ipsu)` — парсит входы, грузит группировку из `storage`, выделяет
  **нераспределённых** клиентов и подсказки; возвращает подготовленный словарь
  (включая `period_start`).
- `assign_new(fio→worker)` — сохраняет назначения в постоянное хранилище.
- `all_workers()` / `employee_workers()` — списки соцработников (из реестра / из общей базы
  по должности).
- `generate(prepared, dept_number, zav_fio, out_path, sheets)` — собирает данные
  (`registry_builder.build`), пишет `.ods` (`ods_writer.generate`) и **сохраняет данные
  текущего месяца** (договоры по клиентам + множество id «доп») для отметок «новый»/«пересмотр»
  в следующем месяце.
- Константа `ALL_SHEETS = ("gos", "dop", "dengi", "peresmotr")`.

### Хранилище (`storage.py`)

Тонкая обёртка над `core.db` / `core.storage`:

| Метод | Назначение |
|---|---|
| `load_settings` / `save_settings` | `reestr/settings.json` (отделения, директор, зам.). |
| `template_path` | путь к `reestr_template.ods`. |
| `load_worker_map` / `save_worker_map` | группировка `{worker_order, client_worker}` → `reestr_workers`/`reestr_clients` в `app.db`. |
| `load_prev` / `save_prev` | `reestr_kv["prev_month"]` = `{contracts, dop_ids}` — для отметок «новый»/«пересмотр». |
| `load_prev_journal` / `save_prev_journal` | `reestr_kv["prev_journal"]` = журнал договоров прошлого месяца по ФИО — для сравнения новый/пересмотр/снят. |

### Парсер (`parser.py`)

Читает `.xls` через `xlrd`. Колонки находит **по заголовкам** (устойчиво к вариациям
формата), с откатом на индексы по умолчанию.

- `parse_registry(path, kind)` → `RegistryInput` (период, записи, подытоги). Определяет
  период (`_find_period`), строку заголовков (`_find_header_row`), карту колонок
  (`_build_colmap`); фильтрует строки без договора/соцработника/суммы/количества.
- `parse_ipsu(path)` → список `IpsuRecord` (ИПСУ: дата выдачи, № ИПСУ, срок).
- Регексы: `_BIRTH_RE` (дата рождения из ФИО), `_CONTRACT_RE` (номер договора, напр.
  `804-З-09`, берётся **последнее** совпадение, пробелы убираются), `_PERIOD_RE`.

### Модель (`model.py`)

- `ServiceRecord` — строка услуги: ФИО/рождение/адрес, договор/дата/период, количество,
  «по договору»/«начислено»/«оплачено», соцработник.
- `IpsuRecord` — запись ИПСУ.
- `RegistryInput` — `kind` (`gos`/`dop`), отделение, период, записи, подытоги.

### Сборщик (`registry_builder.py`)

`build(gos, dop, ipsu, worker_map, worker_order, prev_contracts, prev_dop_ids)` —
группирует по соцработникам и считает итоги:

- **«Гос_»**: клиенты с гос-услугами по соцработникам; сумма «доп» по клиенту; клиенты
  только с «доп» помечаются «Д.»; алфавитная сортировка; итоги по соцработнику и общий.
- **«Доп»**: плоский список доп-услуг; отметка **«новый»** (`client_id ∉ prev_dop_ids`);
  отметка **«пересмотр»** (`_is_peresmotr`); общий итог + сумма прописью.
- **«пересмотр»**: все клиенты по соцработникам с данными ИПСУ (последняя по дате выдачи).
- Приоритет назначения: `worker_map` (постоянная база) выше колонки реестра; иначе
  `«(не распределённые)»`.
- `_is_peresmotr` — сигналы: (1) у ИПСУ несколько программ, у последней иная дата окончания;
  (2) номер договора изменился относительно прошлого месяца.
- «Бесплатно» — `начислено < 0.005`.

### Запись `.ods` (`ods_writer.py` + `ods_build.py`)

- `ods_build.py` — низкоуровневые утилиты odfpy: **логическая адресация** строк/колонок с
  разворачиванием `number-rows/columns-repeated`, клонирование поддерева строки (узлы odfpy
  нельзя `deepcopy` — держат `ownerDocument`/ссылки на соседей), кодирование пробелов
  через `<text:s>`, `set_value`/`put_row`.
- `ods_writer.py` — на каждый лист клонирует **прототипные строки** шаблона и вставляет их
  перед «хвостом»-подписью (паттерн `_stamper`). Раскладка по координатам ячеек для каждого
  из 4 листов; суммы прописью через `num2words_ru`.

### Числа прописью (`num2words_ru.py`)

`rubles_kopecks_in_words(amount)` — рубли словами (с правильным склонением «рубль/рубля/
рублей») + копейки цифрами; `int_in_words(n, female)` — целое прописью с учётом рода.

> Эти же модули (`ods_build`, `num2words_ru`) переиспользует функция «Проезд».

## Журнал договоров и отметки «новый/пересмотр»

Источник отметок в колонке «примечание» (новый/пересмотр) — два механизма:

1. **Журнал договоров** (`journal.py`, необязательный вход). «Отчёт по количеству
   заключённых договоров» — плоский список всех договоров за месяц (и гос, и доп):
   ФИО, тип договора (`_kind` → gos/dop), № договора, период, сумма за месяц.
   `parse_journal` группирует по **нормализованному ФИО** (`norm_fio`), `diff_marks`
   сравнивает с журналом прошлого месяца (`reestr_kv["prev_journal"]`):
   - **новый** — у клиента появился доп-договор;
   - **пересмотр** — у существующего клиента сменился номер гос-договора;
   - **снят** — клиент прошлого месяца отсутствует в текущем (информационно).
   На **первом** запуске журнала сравнивать не с чем — авто-отметки не ставятся, журнал
   просто сохраняется как «прошлый» для следующего месяца. Сопоставление с реестром — по
   ФИО (номер строки `client_id` между месяцами нестабилен, поэтому отметки ведём по ФИО).

2. **Ручная пометка «пересмотр»** (`PeresmotrDialog`) — окно со списком клиентов и
   галочками (предзаполнено отметками из журнала); пользователь правит вручную.

Итоговые множества нормализованных ФИО передаются в `registry_builder.build`
(`mark_new_fios` / `mark_peresmotr_fios`); если заданы — **переопределяют** авто-эвристику
`_is_peresmotr` и пометку «новый». Без журнала и без ручной правки поведение прежнее.

## Проверка (self-test)

```
python main.py --selftest-reestr <папка с входными файлами>
```

Корневая папка `Реестрыы/` — готовый набор входов (`_054`, `_055`, `402`). Эталон:
`Реестрыы/РЕЕСТР май.ods` (сверка должна давать совпадение каждой заполненной ячейки на всех
4 листах).

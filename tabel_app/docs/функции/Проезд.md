# Функция «Проезд»

Формирует помесячный **пакет компенсации проезда** на одного сотрудника в формате `.ods`
из двух документов:

- **маршрутный лист** («отчёт_по_проезду») — список поездок (дата, маршрут откуда→куда,
  цель, № и серия билета);
- **заявление** — обращение к директору с суммой к возмещению **цифрами и прописью**.

Запись `.ods` — через `odfpy`, Excel не требуется. Архитектура — MVC (см.
[АРХИТЕКТУРА.md](../АРХИТЕКТУРА.md)): View=`gui.py`, Controller=`service.py`,
Model=`parser.py`/`ods_writer.py`/`storage.py`.

- **Пакет:** `app/features/proezd/`
- **Ключ функции:** `proezd`
- **Вход:** файл поездок (`.ods/.xls/.xlsx`) + опционально скан билетов (`.jpg/.png`).
- **Выход:** `…ods` с двумя листами (маршрутный лист + заявление).

---

## Поток данных

```
файл поездок ─► parser.read_trips ─┐
скан билетов ─► parser.ocr_tickets ─┤ service.prepare() → {trips, header, tickets}
                                    ▼
       выбор месяца/года в окне → service.build_rows() → строки таблицы (правка в GUI)
                                    ▼
                       service.generate(rows, …) → ods_writer.generate() → пакет.ods
```

1. **Поездки берутся из предоставленного файла** (дата/откуда/куда/цель).
2. **Месяц/год выбираются в окне**, предзаполняются из шапки файла, изменяемы.
3. **Даты проставляются по рабочим дням месяца** по порядку (см. «Правило среды»).
4. **Билеты привязываются 1-й→1-я поездка**; серия нормализуется к известной, цена — по серии.
5. Пользователь **редактирует таблицу целиком** (все колонки + добавление/удаление строк).
6. `.ods` формируется из строк таблицы; сумма = Σ цен.

## Правило рабочих дней («среда — выходной с компенсацией»)

Поездки совершаются по рабочим дням **пн/вт/чт/пт** (не праздник). **Среда** —
методический (выходной) день, **КРОМЕ** случая, когда на её неделе выпал рабочий день
(один из пн/вт/чт/пт стал нерабочим из-за праздника) — тогда среда становится рабочей и
компенсирует его. Правило **автоматическое** (по производственному календарю), ручная
пометка не нужна. Пример: май 2026 — праздник 11.05 (Пн), поэтому среда 13.05 рабочая.

- `service.travel_days(year, month, cal)` — даты-поездки месяца по правилу выше
  (`_wed_is_working` проверяет неделю среды). Дополнительно: явно помеченные перенесённые
  рабочие дни (`calendar_years.work_days`, напр. рабочая суббота) включаются всегда.
- **Группировка строк по дням (по структуре отчёта):** один день = **2 основные поездки +
  прилегающие к ним допы**. Допы помечены **голубым** в исходном файле; `parser.read_trips`
  читает цвет ячеек и ставит `trip["dop"]=True` для голубых (`_is_cyan`). `service._day_groups`
  считает основные (не-доп) строки: каждые 2 основные с относящимися допами = группа-день.
- `service.assign_dates(trips, year, month, cal)` — проставляет датам рабочие дни месяца
  **по группам, с нуля** (не глядя на текущее содержимое колонки): группа-день `i` → `i`-й
  рабочий день месяца. Формат — `ДД.ММ.ГГГГ`. При несовпадении числа групп и рабочих дней —
  проставляет сколько возможно и предупреждает. Признак `dop` хранится в **скрытой колонке**
  таблицы (`gui`), поэтому кнопка работает и после правок (добавленные вручную строки —
  основные).
- `service.reassign_dates(rows, year, month)` — то же для текущих строк по кнопке
  «Проставить даты по месяцу» (работает даже при пустой колонке «Дата»).
- Источник переноса — **общий производственный календарь (БД)**; правится в «Табель» →
  «Производственный календарь…» → поле «Перенесённые РАБОЧИЕ дни».

## Слои

### View (`gui.py`) — `ProezdWindow`

- Выбор файла поездок и скана; просмотр скана.
- **Поля «Месяц»/«Год»** — предзаполняются из шапки файла, изменяемы; кнопка **«Проставить
  даты по месяцу»** пересчитывает даты по выбранному периоду.
- Поля ФИО: **родительный падеж** и **краткая форма** «Фамилия И.О.» (автосклонение, правятся).
- **Таблица** с колонками `№ · Дата · Откуда · Куда · Цель · № билета · Серия · Цена`:
  редактируется **любая** ячейка (двойной клик), при смене серии цена пересчитывается;
  кнопки **«+ Строка»/«− Строка»** (вставка после выделенной) с авто-перенумерацией.
  Внизу — итоговая сумма.
- **Подсказки (автодополнение).** При вводе в ячейку (откуда/куда/цель/серия/№ билета)
  выпадает список знакомых **целых значений** по подстроке: навигация ↑/↓, выбор —
  Enter или клик. Источник — лексикон (`service.build_lexicon`): текущая таблица +
  накопленная история (`lexicon.json`, пополняется при формировании) + ФИО клиентов из
  общей базы (Реестр). См. `gui._begin_edit`/`_column_pool`.
- **Enter → ячейка ниже.** Если список подсказок не открыт, Enter сохраняет ячейку и
  переходит к редактированию той же колонки в следующей строке (быстрый ввод столбца).
- Кнопки: **«Прочитать»** (`_read` → `service.prepare`+`build_rows`), **«Сформировать
  (.ods)»** (`_generate` → `service.generate`).
- View **тонкий**: вся логика (даты, серии, цена, запись) — в `service`.

### Controller (`service.py`)

- `prepare(trips_path, scan_path)` → `{trips, header, tickets, ocr_available}` (даты тут не
  ставятся — это делает `build_rows` по выбранному месяцу).
- `build_rows(prepared, year, month, settings)` → `(rows, note)` — строки таблицы (ключи
  `ROW_KEYS`: num/date/frm/to/purpose/number/series/price): даты по месяцу, серия
  нормализована, цена по серии.
- `travel_days` / `assign_dates` / `reassign_dates` — генерация/пересчёт дат (см. выше).
- `normalize_series(raw, settings)` — привязка распознанной серии к известной
  (`settings.known_series`) по совпадению **цифрового хвоста** (напр. `ИН-575` → `4МН-575`),
  при неоднозначности — по близости (Левенштейн); иначе — нормализованная распознанная.
- `default_price_for(series, settings)` — цена по префиксу серии (`4МН`→35, `ВА`→43), иначе
  `default_price`.
- `build_ctx(header, year, month, overrides)` — контекст заявления/отчёта (ФИО в формах,
  месяц прописью/предложный, год, реквизиты, дата заявления = 1-е число следующего месяца).
- `generate(rows, header, year, month, out_path, overrides)` → `ods_writer.generate` из строк
  таблицы; сумма = Σ цен.

### Model — данные/IO

**`storage.py`:**
- `load_settings()` — пользовательский `settings.json`, **дополненный новыми ключами из
  пакетного дефолта** (правки пользователя в приоритете). Решает проблему «JSON — только
  сид»: новые ключи (напр. `known_series`) появляются и на старых установках.
- `load_calendar()` → `ProductionCalendar` из общей БД (`work_days` учитывается).
- `template_path`; `load/save_worker_forms`, `forms_for`, `remember_forms`; **автосклонение
  ФИО** (`derive_genitive`/`derive_short`, пол по отчеству).

**`parser.py`:**
- `read_trips(ods_path)` → `(trips, header)`; `read_tickets` — резерв из файла.
- OCR (best-effort, офлайн, Windows OCR): `ocr_available`, `ocr_tickets` — сегментация
  `_segment_tickets`, повороты 0/90/–90°, бинаризация, `_parse_ticket`. **Серия** читается
  надёжно (нормализуется в `service.normalize_series` к известной); **номер** —
  best-effort (матричный шрифт, часто пусто → вписывается вручную). При недоступности OCR → `[]`.

**`ods_writer.py`:**
- `generate(template, out, ctx, rows)` — листы «отчёт_по_проезду» и «Заявление»; отчёт
  клонирует прототипную строку на каждую строку таблицы; сумма = Σ `price`.
- `_write_report` / `_write_application`; использует `reestr.ods_build` и
  `reestr.num2words_ru` (сумма/количество прописью).

## Настройки (`data/settings.json`)

Кроме реквизитов: `default_price`, `series_prices` (префикс→цена), **`known_series`**
(полные известные серии для привязки OCR, напр. `["4МН-571","4МН-575","ВА-996"]`),
`months_upper`/`months_prep`.

## Примечания

- Шаблон: `proezd/templates/proezd_template.ods`.
- Корневая папка `Проезд/` — примеры сканов и образцы `.ods`.
- Каталог `neuro/` (CRNN на PyTorch для точечных **номеров**) — эксперимент, **в приложение
  не подключён**; рабочий OCR — Windows OCR в `parser.py`.
- В **лёгкой** сборке функция «Проезд» отсутствует (исключается вместе с `cv2`/`winrt`).

# Обзор приложения

Настольное Windows-приложение (Tkinter) для отдела соцобслуживания: одно окно-лаунчер
(главное меню) открывает независимые **функции**, каждая из которых формирует готовый
офисный документ. **Microsoft Excel и любой офисный пакет не требуются** — все писатели
на чистом Python (`xlwt` для `.xls`, `odfpy` для `.ods`, `xlrd` для чтения шаблонов/входов).

Весь код и интерфейс — на русском; эта конвенция сохраняется.

## Функции

| Функция | Что делает | Формат | Документ |
|---|---|---|---|
| **Табель** | табель учёта рабочего времени (форма Т-13) | `.xls` | [функции/Табель.md](функции/Табель.md) |
| **Реестр** | реестр оплаты услуг (4 листа) из реестров гос/доп + ИПСУ | `.ods` | [функции/Реестр.md](функции/Реестр.md) |
| **Приложение к табелю** | нагрузка чел/день на соцработника (город/частный) с перераспределением | `.xls` | [функции/Приложение.md](функции/Приложение.md) |
| **Проезд** | пакет компенсации проезда (маршрутный лист + заявление) | `.ods` | [функции/Проезд.md](функции/Проезд.md) |
| **Услуги-Деньги** | отчёт по услугам и деньгам из 071 + списков льготников (шаблон с формулами) | `.xlsx` | [функции/Услуги-Деньги.md](функции/Услуги-Деньги.md) |
| **График проверок** | график проверок качества работы соцработников на полугодие (ротация) | `.xlsx` | [функции/Графики.md](функции/Графики.md) |
| **Протокол** | протокол рабочего совещания (методчаса) отделения по шаблону | `.odt` | [функции/Протокол.md](функции/Протокол.md) |

**Главное меню** прокручиваемое (Canvas + колесо мыши) и содержит кнопку **«Сохранённые
документы»**. Все сформированные файлы (любой функции) **дублируются копией в базу** (`documents`)
и доступны в окне архива (`core/documents.py`, `core/documents_gui.py`) — открыть/сохранить/удалить.

Архитектура (слои MVC, SOLID, подсистемы, сборка версий) — опорный документ
[АРХИТЕКТУРА.md](АРХИТЕКТУРА.md). База данных — [БАЗА_ДАННЫХ.md](БАЗА_ДАННЫХ.md).

Приложение поставляется в **двух версиях**: «Полная» (все функции, с OCR) и «Лёгкая»
(без «Проезда» и без `cv2`/`winrt`, в разы меньше). См. §«Сборка».

## Архитектура

```
main.py ──► app/shell.py (главное меню: карточка на функцию)
                │  get_features()  ◄── app/features/registry.py
                ▼
   ┌──────────────────────────────────────────────┐
   │  app/features/<name>/                          │
   │    __init__.py  FEATURE_KEY/TITLE/DESCRIPTION  │
   │    gui.py       open_<name>(master)→Toplevel   │
   │    service.py   оркестрация                     │
   │    storage.py   тонкая обёртка над core         │
   │    (calc/parser/model/…)  логика функции        │
   │    writer (excel_writer.py | ods_writer.py)     │
   └──────────────────────────────────────────────┘
                │
                ▼
   app/core/    db.py       единая SQLite-база app.db
                storage.py  пути (source ↔ frozen), JSON I/O
                xls_util.py  палитра RGB + перенос стилей шаблона
```

### Лаунчер и контракт функции

- `app/shell.py` (`Shell(tk.Tk)`, `run()`) рисует по карточке на функцию из
  `get_features()`; кнопка «Открыть» вызывает `feat.open(master)`.
- `app/features/registry.py` — класс `Feature(key, title, description, opener)` и список
  `get_features()`. Функция регистрируется **одной строкой** `Feature(...)`.
- Контракт функции:
  - `app/features/<name>/__init__.py` экспортирует `FEATURE_KEY`, `FEATURE_TITLE`,
    `FEATURE_DESCRIPTION`;
  - `app/features/<name>/gui.py` экспортирует `open_<name>(master) -> tk.Toplevel`.
- **Добавить функцию:** создать пакет, реализовать опенер, добавить строку `Feature(...)`
  в `registry.py` и `--add-data` в `build.ps1` для ресурсов `data/`/`templates/`
  (назначение `features/<name>/...`). Подробности — `tabel_app/README.md`.

### Общее ядро (`app/core/`)

- **`db.py`** — единая SQLite-база `app.db`, единственный источник правды для всех функций
  (справочник + таблицы `pril_*` + `reestr_*`). Засев/миграция идемпотентны и под флагами в
  `meta`. Подробно — [БАЗА_ДАННЫХ.md](БАЗА_ДАННЫХ.md).
- **`storage.py`** — работа из исходников **и** из собранного `.exe`:
  - пользовательские данные → `<рядом с программой>/data/<feature>/`
    (`app_base_dir()`, `feature_data_dir()`);
  - встроенные ресурсы (шаблоны, JSON по умолчанию) → `_MEIPASS/features/<feature>/` во
    frozen, иначе пакет в исходниках (`feature_resource_dir()`, `template_path()`);
  - `load_json` (с разворачиванием default-файла при первом запуске), `save_json`
    (атомарная запись через временный файл, UTF-8 с кириллицей).
- **`xls_util.py`** — общие помощники `.xls`:
  - `ColourPalette` — аллокатор произвольных RGB в палитру `.xls` (слоты 0x20…0x3F,
    ближайший цвет при переполнении);
  - `TemplateStyles` — перенос форматирования из шаблона (`xlrd`, `formatting_info=True`)
    в `xlwt.XFStyle` (шрифт/выравнивание/границы/заливка/числовой формат), с кэшем;
  - `col_width`, `row_height`, `thin_borders`, `solid_fill`.

### Писатели документов (без Excel)

Каждый писатель **тиражирует прототипные строки шаблона** до нужного количества с
сохранением форматирования, затем заполняет данные. Раскладка задана по координатам ячеек.

- **Табель** (`excel_writer.py`, `xlwt`) — читает `t13_template.xls` через `xlrd`,
  транслирует стили (`TemplateStyles`), копирует заголовок 1:1, реплицирует пару строк
  сотрудника, сдвигает подписи на `2*(N-1)`; синяя заливка выходных через `ColourPalette`.
- **Приложение** (`excel_writer.py`, `xlwt`) — строится **с нуля** (Calibri 11), стили
  кэшируются по сигнатуре, объединения `write_merge`, серая заливка выходных `0xE6E6E6`,
  альбомная печать «в 1 страницу».
- **Реестр** (`ods_writer.py`, `odfpy`) — загружает `reestr_template.ods`, клонирует
  прототипные `<table:table-row>` на каждом из 4 листов и вставляет перед хвостом-подписью.
  Логическая адресация строк/колонок и клонирование поддеревьев — в `ods_build.py`.
- **Проезд** (`ods_writer.py`, `odfpy`) — загружает `proezd_template.ods`, пишет два листа
  (маршрутный лист + заявление); переиспользует `reestr/ods_build.py` и
  `reestr/num2words_ru.py` (суммы прописью).

> **Межфункциональная зависимость:** «Проезд» импортирует `ods_build` и `num2words_ru` из
> пакета «Реестр» — это общие ODS-утилиты, не отдельный модуль ядра.

## Сборка

```
cd tabel_app
powershell -ExecutionPolicy Bypass -File build.ps1                 # Полная (4 функции, OCR)
powershell -ExecutionPolicy Bypass -File build.ps1 -Variant lite   # Лёгкая (без Проезда/OCR)
```

`build.ps1` собирает `dist\Tabel.exe`. Параметр `-Variant lite` исключает пакет
`app.features.proezd` и тяжёлые `cv2`/`winrt`/`numpy` (мягкая регистрация в `registry.py`
сама пропустит «Проезд»). Финальная упаковка в **папку-комплект** (`Табель.exe` + `data/` +
`Инструкция.txt`) и `.zip` делается отдельно (имена с кириллицей — вне ASCII-скрипта).
Готовые архивы — в корневой папке `Сборки/`.

**Установщики (Inno Setup).** Скрипты `tabel_app/installer/{full,lite}.iss`, компилятор
`ISCC.exe` (Inno Setup 6, через winget). Ставят в `%LOCALAPPDATA%\Tabel` (полная) /
`…\Tabel-lite` (лёгкая) **без админ-прав**, с ярлыками (рабочий стол + Пуск) и
деинсталлятором; `data` ставится только при отсутствии (не затирает БД при обновлении).
Готовые `…setup.exe` — в корневой папке `Установщики/`.

**Обратная связь.** В каждом окне функции есть кнопка **«Отзывы, жалобы и пожелания»**
(общий компонент `app/core/feedback.py`): кнопка «Отправить» шлёт письмо разработчику на
`farcrystas@gmail.com` **прямо из приложения** через сервис **Web3Forms** (HTTPS, без
своего сервера и без паролей в `.exe`), добавляя служебные данные (функция, версия, ОС).
Если нет интернета — запасной путь: почтовая программа или «Скопировать».

- PyInstaller ломается на **кириллице в пути** проекта, поэтому `build.ps1` `robocopy`-ит
  исходники в ASCII-каталог `C:\Users\<user>\tabel_build`, собирает там и выдаёт
  `dist\Tabel.exe` (для поставки переименовать в `Табель.exe`).
- **`build.ps1` держать только в ASCII** — Windows PowerShell 5.1 неверно читает UTF-8 без BOM.
- Ключевые опции: `--windowed --onefile --name Tabel --collect-submodules app`, набор
  `--add-data` для шаблонов/JSON каждой функции (назначение `features/<name>/...`), скрытые
  импорты `xlrd`/`sqlite3`, сбор подмодулей `xlwt`/`odf`.

## Проверка (нет тест-фреймворка)

Валидация — через скрытые self-test'ы в `main.py` (у windowed-`.exe` нет консоли, поэтому
каждый пишет результат/трассировку в `.txt` рядом с программой):

| Команда | Что делает | Эталон для сверки |
|---|---|---|
| `python main.py --selftest` | строит табель | корневой `табель май.xls` |
| `python main.py --selftest-reestr <папка>` | строит реестр из папки входов | `Реестрыы/РЕЕСТР май.ods` |
| `python main.py --selftest-pril` | засев БД + приложение | `Приложение_9_Июнь_2026.xls` |
| `python main.py --selftest-proezd` | проезд из шаблонных билетов | образцы в `Проезд/` |

Каждый self-test пишет `selftest*_result.txt` (строка `OK <путь>` или трассировка). Готовый
набор входов реестра — корневая папка `Реестрыы/`.
